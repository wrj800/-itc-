# 优秀项目对标与改进方案

本文档用于把优秀开源项目的工程化经验，转化为本毕业设计项目可以执行的改进路线。目标不是盲目堆技术，而是让项目同时具备“能运行、能解释、能维护、能验证、能答辩”的质量。

## 一、对标项目和学习重点

| 对标对象 | 学习重点 | 本项目对应改进 |
|---|---|---|
| Spring PetClinic | 小而完整的 Spring Boot 样例、清晰领域模型、测试和本地运行说明 | 后端模块拆分、领域对象持久化、接口测试 |
| JHipster | 前后端分离、权限、Docker、CI、OpenAPI、工程脚手架完整度 | 登录权限、数据库迁移、CI、环境一键启动 |
| Vue Vben Admin / SoybeanAdmin | Vue 3、TypeScript、后台布局、路由、权限菜单、组件化 | 拆分 `App.vue`、建立路由、菜单权限、统一布局组件 |
| Prometheus 官方实践 | 监控目标、告警规则、指标查询、可观测性思路 | 监控采集、告警规则、真实指标和运维闭环 |
| GitHub Open Source Guides | README、贡献说明、Issue/PR 模板、可维护协作流程 | 补齐开源项目标准文件和开发交接规范 |

## 二、优秀项目应具备的核心特征

### 2.1 一眼能看懂

优秀项目的 README 应该让第一次打开仓库的人快速知道：

1. 这个项目解决什么问题。
2. 技术栈是什么。
3. 如何启动。
4. 当前已经完成什么。
5. 如何验证功能是否正常。
6. 后续计划是什么。

本项目当前 README 已有基础结构，但还需要继续增强截图、架构图、核心流程、演示账号、常见问题和答辩亮点。

### 2.2 运行路径稳定

优秀项目不会只写“理论上可以运行”，而是提供明确的启动脚本、环境要求和健康检查。

本项目需要保持以下验证边界：

1. Docker 观测栈是否启动。
2. Prometheus 是否能查询 `docker` 和 `wsl` 两个数据源。
3. 后端 `/api/health` 是否返回 200。
4. 前端是否能访问。
5. AI 服务是否能返回诊断结果。
6. 数据库初始化是否成功。

### 2.3 代码结构清晰

优秀项目通常有清楚的分层：

```text
controller -> service -> repository -> domain/entity
view/page -> component -> api/client -> store/router
```

本项目下一阶段必须优先拆分：

1. 后端从内存样例服务转为 `domain`、`service`、`repository` 分层。
2. 前端从单个 `App.vue` 拆分为布局、页面、业务组件和接口层。
3. AI 服务从规则化返回逐步拆分为检索、提示词、模型调用、结果解析。

### 2.4 文档和代码同步

优秀项目的文档不是摆设。新增接口、数据库表、配置项、运行方式，都必须同步说明。

本项目已经要求更新：

```text
docs/开发文档.md
docs/开发记录.md
docs/开发过程记录.md
docs/设计图索引.md
```

后续还应补充：

1. `docs/API接口规范.md`
2. `docs/数据库设计.md`
3. `docs/测试计划.md`
4. `docs/演示与答辩说明.md`

### 2.5 有自动化质量门禁

优秀项目不会只靠手工检查。每次提交至少要自动执行：

1. 后端测试。
2. 前端构建。
3. 关键脚本语法检查。
4. 文档链接和核心文件存在性检查。

本阶段已新增 GitHub Actions 工作流，先执行 Maven 测试和前端构建。后续可继续补充单元测试、接口测试、E2E 测试和 Docker Compose 启动验证。

## 三、本项目质量分级目标

### 3.1 基础合格

1. 项目能启动。
2. 前端能访问。
3. 后端接口能返回。
4. README 能说明基本启动方式。
5. 文档和代码不明显脱节。

当前项目已经达到这个阶段。

### 3.2 毕设优秀

1. 有真实可运行的监控采集链路。
2. 有数据库持久化。
3. 有管理员和维护人员角色权限。
4. 有告警转工单和工单状态流转。
5. 有 AI 辅助诊断闭环。
6. 有 UML、ER 图、部署图和流程图。
7. 有测试和 CI。
8. 有演示脚本和答辩材料。

这是当前主要目标。

### 3.3 开源项目质量

1. 有完整贡献说明。
2. 有 Issue 和 PR 模板。
3. 有清晰的 Roadmap。
4. 有统一代码规范和提交规范。
5. 有接口文档和演示数据。
6. 能通过 Docker Compose 一键启动完整环境。
7. 有较完整的单元测试、集成测试和前端测试。

这是毕业设计完成后的加分方向。

## 四、阶段改进路线

### 阶段一：工程化底座

目标：让仓库像一个成熟项目。

任务：

1. 补充 `CONTRIBUTING.md`。
2. 补充 Issue 模板和 PR 模板。
3. 补充 GitHub Actions CI。
4. 在开发规则中加入提交规范、测试规范和 PR 检查项。
5. README 增加项目质量说明和验证入口。

### 阶段二：前端工程化

目标：解决 `App.vue` 过大和后续不可维护问题。

任务：

1. 引入 Vue Router。
2. 拆分 `layouts/AdminLayout.vue`、`layouts/MaintainerLayout.vue`。
3. 拆分 `pages/admin/*` 和 `pages/maintainer/*`。
4. 拆分 `components/common/*`，包括 KPI 卡片、数据表格、看板列、详情面板。
5. 建立 `services/api.ts`，统一请求后端接口。

### 阶段三：真实权限与数据库

目标：让角色分离从前端演示变成真实系统能力。

任务：

1. 新增用户、角色、权限、菜单表。
2. 新增登录接口和 JWT。
3. 前端登录后根据角色进入不同工作台。
4. 接口按角色校验权限。
5. 工单与处理人、责任组关联。

### 阶段四：可观测性闭环

目标：让“真实运维”成为项目核心亮点。

任务：

1. 将 Prometheus 查询结果沉淀为监控目标状态。
2. 建立告警规则和告警事件模型。
3. 支持 Blackbox 探测失败转告警。
4. 支持 node_exporter 指标异常转告警。
5. 支持告警转工单、工单处置、关闭和复盘。

### 阶段五：AI 诊断增强

目标：让 AI 不只是文本生成，而是基于上下文辅助排障。

任务：

1. FastAPI 接收告警、指标、日志和工单上下文。
2. 向量数据库保存知识库片段、历史工单和处置经验。
3. 检索相关知识后调用大模型 API。
4. 返回可能原因、排查步骤、风险等级和回滚建议。
5. 将诊断结果写回工单处理记录。

## 五、细节注意事项

1. 不要为了显得复杂而加入暂时无法验证的技术。
2. 每个新增功能必须能在前端、后端、文档、测试中形成闭环。
3. 页面不要只有静态卡片，要能体现真实数据来源和业务动作。
4. 接口不要只返回固定假数据，至少要逐步迁移到数据库或真实 Prometheus 查询。
5. AI 诊断必须说明输入上下文、输出结构和人工确认边界。
6. 演示环境要稳定，答辩前优先保证核心链路可运行。
7. 每次改动后必须写清楚验证结果，区分“命令未启动”“命令启动但失败”“命令通过但有警告”。

## 六、近期最应该做的事

当前最应该优先做：

1. 完成仓库工程化文件和 CI。
2. 拆分前端组件和路由。
3. 实现真实登录与角色权限。
4. 设计并落地 MySQL 核心表。

原因：前端已经具备视觉展示能力，但代码集中在 `App.vue`，且登录仍是演示态。如果继续堆页面，后续维护成本会快速上升。先把工程结构和权限模型打稳，后面的资产、告警、工单、AI 诊断才容易扩展。
