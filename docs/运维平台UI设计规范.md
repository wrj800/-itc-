# 运维平台 UI 设计规范

本文档用于约束 ICT 智能运维平台后续所有页面和组件设计。目标是让系统接近先进大型运维平台、NOC 指挥平台和企业级后台，而不是普通演示页。

## 一、总体设计原则

1. **工作台优先**：首屏必须直接进入真实业务，不做营销式首页。
2. **状态先行**：运维界面先展示系统健康、风险、告警、SLA，再展示装饰性内容。
3. **密度适中**：信息要可扫描，但不能拥挤到影响判断。
4. **角色隔离**：管理员看全局、配置和治理；维护人员看待办、处置和记录。
5. **真实来源**：重要指标要能说明来自 Prometheus、Blackbox、node_exporter、工单系统或数据库。
6. **可行动**：告警、工单、诊断建议旁边必须有明确操作，例如接单、转派、提交处置、关闭。
7. **少解释多呈现**：页面内不写大段说明，不把功能介绍当作 UI。

## 二、布局规则

### 2.1 管理员后台

管理员后台采用：

```text
深色侧边栏 + 顶部工具栏 + 页面标题 + 指挥状态条 + KPI + 图表/表格/看板
```

首屏优先顺序：

1. 指挥状态条：采集状态、告警管道、SLA 风险、最近同步。
2. KPI：在线设备、活跃告警、今日工单、服务可用率、自动化成功率。
3. 核心服务健康度。
4. 告警趋势和告警级别分布。
5. 最新告警和待处理工单。

禁止：

1. 首屏放大幅宣传图。
2. 使用过多大卡片导致信息稀疏。
3. 在管理端展示维护人员才需要的细碎处置记录。

### 2.2 维护人员工作台

维护人员工作台采用：

```text
班组值班状态 + 我的 KPI + 工单看板 + 右侧工单详情/AI 诊断/处理记录
```

首屏优先顺序：

1. 当前班组、负责范围、交接状态、升级通道。
2. 待处理工单、即将超时、今日已完成、平均响应。
3. 我的工单看板。
4. 工单详情、SLA、故障描述。
5. AI 诊断建议和处置按钮。
6. 处理记录。

禁止：

1. 维护人员端出现系统设置、用户权限、全局规则配置。
2. 右侧详情只有输入框，没有工单上下文。
3. KPI 文案竖排、数值断行、按钮文字挤压。

## 三、组件规则

### 3.1 KPI 卡片

每张 KPI 卡必须包含：

1. 图标。
2. 指标名称。
3. 当前值。
4. 与昨日或基线对比。
5. 小趋势线或状态色。

规则：

1. 数字必须醒目，但不能使用英雄级字号。
2. 单位要短，例如 `%`、`分钟`、`Mbps`。
3. 指标名最多两行，不能竖排。
4. 红色只用于危险或负向状态。
5. 管理端 KPI 关注全局，维护端 KPI 关注个人待办。

### 3.2 指挥状态条

指挥状态条用于表达系统运行态，不替代 KPI。

每项包含：

1. 状态名称。
2. 当前状态值。
3. 一句短描述。
4. 左侧状态色条。

推荐内容：

1. 采集状态。
2. 告警管道。
3. SLA 风险。
4. 最近同步。

### 3.3 表格

表格用于承载可比较的数据。

规则：

1. 表头固定语义清晰。
2. 状态列必须使用标签。
3. 操作列不堆过多文字，后续应改为按钮或下拉菜单。
4. 长文本必须截断或进入详情，不撑破列宽。
5. 每页表格应配筛选、搜索或分组入口。

### 3.4 告警组件

告警必须包含：

1. 告警级别。
2. 告警名称。
3. 来源对象。
4. 触发指标。
5. 当前状态。
6. 处理人或责任组。
7. 发生时间或持续时间。

告警级别颜色：

| 级别 | 颜色语义 |
|---|---|
| P1 / 紧急 | 红色 |
| P2 / 重要 | 橙色 |
| P3 / 一般 | 蓝色 |
| 提示 | 绿色或灰色 |

### 3.5 工单看板

看板列建议：

```text
待接单 / 待分派
处理中
待验证
已完成
```

工单卡片必须包含：

1. 标题。
2. 工单编号。
3. 摘要。
4. 处理人。
5. 类型或标签。
6. SLA 或优先级。

### 3.6 工单详情面板

详情面板必须像真实工单，而不是普通表单。

必须包含：

1. 工单编号。
2. 优先级。
3. 状态。
4. SLA 剩余时间。
5. 关联告警。
6. 告警来源。
7. 关联设备。
8. 故障描述。
9. AI 诊断建议。
10. 处置动作。
11. 处理记录。

### 3.7 AI 诊断组件

AI 诊断不能只显示一个输入框。

必须包含：

1. 当前故障上下文。
2. 输入或补充现象。
3. 生成建议按钮。
4. 诊断结论。
5. 推荐动作。
6. 人工确认或写入工单动作。

### 3.8 登录入口与表单组件

管理员登录和维护人员登录必须使用同一套表单栅格，不能因为角色不同导致卡片高度、输入框宽度或按钮位置跳动。

规则：

1. 两种登录角色必须保持相同字段数量、相同行高、相同控件高度。
2. 角色差异只能体现在字段文案、选项内容和提交动作，不允许一方多一行字段、一方少一行字段。
3. 表单字段统一采用“字段名 + 控件”的单列结构，不能把短文本强行拆成两列。
4. `input`、`select`、主按钮高度必须一致，当前基线为 `52px`。
5. 标题区、表单区、选项区和提交按钮区必须有稳定高度，切换角色时不能上下抖动。
6. 复选框行必须单独定义样式，不能继承普通表单字段的网格规则。
7. 按钮、字段标签、选项文字默认不换行，窄屏时通过整体布局降级处理。
8. 登录页修改后必须同时检查 `?loginRole=admin` 和 `?loginRole=maintainer`。

### 3.9 工具栏与动作按钮

顶部工具栏、页面动作区和业务动作区的按钮必须稳定，不能因为容器宽度变化导致短文案断行。

规则：

1. `刷新`、`导出报表`、`创建工单`、`提交处置`、`转入我的工单` 等按钮文案默认单行显示。
2. 图标和文字之间保持固定间距，不能重叠。
3. 工具栏按钮必须设置最小宽度和 `white-space: nowrap`。
4. 页面动作按钮必须与标题区垂直居中对齐。
5. 窄屏时可以整体换行到下一行，但按钮内部文字不能上下拆分。

## 四、视觉规则

1. 卡片圆角不超过 8px。
2. 页面背景保持浅灰，不使用大面积渐变装饰。
3. 深色只用于侧边栏、状态强调和必要的运维氛围。
4. 主色使用青绿色，辅助色使用蓝、橙、红、灰。
5. 不使用单一紫色、深蓝或橙棕色铺满页面。
6. 页面文字不能和图标、按钮、卡片重叠。
7. 按钮文案必须完整显示。
8. 所有固定格式组件要有稳定尺寸，避免 hover 或数据变化导致布局抖动。

## 五、交互规则

1. 刷新按钮必须对应真实数据刷新。
2. 告警转工单必须避免重复创建。
3. 工单操作必须形成状态流转。
4. 维护人员只看到自己相关任务。
5. 管理员才能看到全局配置和报表导出。
6. AI 建议不能自动关闭工单，必须由人工确认。

## 六、截图验收规则

每次视觉修改后至少检查：

```text
http://127.0.0.1:5173
http://127.0.0.1:5173?loginRole=admin
http://127.0.0.1:5173?loginRole=maintainer
http://127.0.0.1:5173?role=admin&view=overview
http://127.0.0.1:5173?role=maintainer&view=myTickets
```

截图尺寸：

```text
1600x900
```

检查项：

1. 是否有文字截断。
2. 是否有按钮挤压。
3. 是否有 KPI 竖排。
4. 是否有明显大空白。
5. 是否有横向溢出。
6. 是否角色功能混淆。
7. 是否核心业务动作在首屏可见。
8. 是否看起来像真实运维平台，而不是静态演示页。

## 七、下一步 UI 改进方向

1. 拆分布局和组件，避免继续堆在 `App.vue`。
2. 用 ECharts 替换 CSS 假趋势图。
3. 表格操作列改为图标按钮和下拉菜单。
4. 增加筛选条、环境选择、时间范围和责任组筛选。
5. 加入真实接口状态、加载态、空状态和错误状态。
6. 设计移动端降级布局，但桌面端仍以高效工作台为主。
