## 二、当前项目定位

项目名称：企业 ICT 智能运维与故障诊断平台。

项目目标：

1. 管理企业 ICT 设备与运维对象。
2. 通过 Prometheus、Exporter、Blackbox Exporter 采集运行状态。
3. 支持真实 Linux 虚拟节点监控，不停留在纯前端页面模拟。
4. 后续接入 AI 服务，实现告警摘要、故障根因分析、处置建议和知识库问答。

## 五、已实现接口

### 5.1 健康检查

```http
GET /api/health
```

作用：检查 Spring Boot 后端是否可用。

### 5.2 Prometheus 查询

```http
GET /api/monitor/prometheus/query?source=docker&query=up
GET /api/monitor/prometheus/query?source=wsl&query=up
```

参数说明：

| 参数 | 说明 |
|---|---|
| `source` | 数据源，当前支持 `docker` 和 `wsl` |
| `query` | PromQL 查询语句 |

### 5.3 Prometheus 目标查询

```http
GET /api/monitor/prometheus/targets?source=docker
GET /api/monitor/prometheus/targets?source=wsl
```

作用：查询 Prometheus 当前采集目标。

### 5.4 运维业务总览

```http
GET /api/ops/overview
```

作用：返回设备资产数量、在线设备数量、活跃告警数、未关闭工单数、监控目标可用率。

### 5.5 设备资产台账

```http
GET /api/ops/assets
```

作用：返回当前运维对象清单，包含服务器、网络链路、BMC 模拟对象等。

当前字段：

| 字段 | 说明 |
|---|---|
| `id` | 设备唯一编号 |
| `name` | 设备名称 |
| `type` | 设备类型 |
| `model` | 型号或平台 |
| `address` | IP、域名或探测地址 |
| `status` | `online`、`warning`、`offline` |
| `location` | 所属位置 |
| `collector` | 采集方式 |
| `lastSeen` | 最近一次发现时间 |

### 5.6 监控目标

```http
GET /api/ops/targets
```

作用：返回平台沉淀后的监控目标列表，供前端展示目标状态和延迟。

### 5.7 告警中心

```http
GET /api/ops/alerts
```

作用：返回当前告警事件列表。

当前字段：

| 字段 | 说明 |
|---|---|
| `id` | 告警编号 |
| `assetId` | 关联设备 |
| `title` | 告警标题 |
| `severity` | 告警级别 |
| `status` | 告警状态 |
| `metric` | 触发指标 |
| `summary` | 告警摘要 |
| `createdAt` | 发生时间 |

### 5.8 告警转工单

```http
POST /api/ops/alerts/{alertId}/work-orders
```

请求体：

```json
{
  "title": "handle edge link probe jitter",
  "category": "network",
  "assignee": "oncall"
}
```

作用：将指定告警转为工单。如果告警已经有关联工单，则直接返回已有工单，避免重复创建。

### 5.9 工单流转

```http
GET /api/ops/work-orders
```

作用：返回当前工单列表，前端按 `open`、`processing`、`closed` 三类状态展示为看板。

### 5.10 AI 故障诊断

```http
POST /api/ops/diagnose
```

请求体：

```json
{
  "question": "blackbox link probe down"
}
```

作用：根据故障描述返回事件级别、可能原因、建议动作和诊断结论。当前为规则化诊断结果，后续接入 FastAPI、大模型 API、向量数据库和知识库上下文。

## 六、当前监控数据源

| 数据源 | 地址 | 用途 |
|---|---|---|
| `docker` | `http://127.0.0.1:9090` | Docker 观测栈 |
| `wsl` | `http://<WSL-IP>:19090` | 真实 Linux 虚拟节点 |

WSL IP 可能变化，`scripts/start-backend.ps1` 会自动设置：

```text
ICT_PROMETHEUS_WSL_BASE_URL=http://<WSL-IP>:19090
```

## 七、当前前端模块

当前 `frontend/src/App.vue` 已改为角色分离式前端：

1. 登录分流页：管理员入口与维护人员入口分离。
2. 管理员后台：包含总览、监控中心、资源管理、告警中心、工单中心、自动化巡检、报表分析、系统设置。
3. 维护人员工作台：包含我的工单、告警待办、设备状态、处置记录。
4. 管理员与维护人员使用不同导航和不同页面职责，避免维护人员看到系统配置、用户权限等管理员功能。
5. 页面风格参考企业级 ICT 运维后台，使用深色侧栏、顶部工具栏、浅色数据卡片、表格、看板和右侧详情面板。

当前前端视觉稿：

```text
docs/设计图/前端视觉稿/01_登录分流页.png
docs/设计图/前端视觉稿/02_管理员后台.png
docs/设计图/前端视觉稿/03_维护人员工作台.png
```

本轮循环自检后的前端约束：

1. 登录页必须同时体现角色分流和真实运维环境，不再只展示表单。
2. 管理员端聚焦全局管理视角，展示资源、告警、工单、服务健康度、趋势和报表入口。
3. 维护人员端聚焦处置视角，展示我的工单、SLA、故障描述、AI 诊断建议、处置动作和处理记录。
4. 管理员预览身份为 `张伟`，维护人员预览身份为 `李四`。
5. 前端视觉稿通过原路径覆盖更新，不保留旧版截图副本。

## 八、后续开发优先级

1. 拆分 `App.vue`，将管理员端、维护人员端、通用 KPI、表格、看板、详情面板拆成独立组件。
2. 引入 Vue Router，为管理员端和维护人员端建立独立路由和路由守卫。
3. 为管理员和维护人员补充真实登录、JWT、角色权限和菜单权限。
4. 将当前内存样例数据迁移为 MySQL 表和 Spring Data/JPA 或 MyBatis 持久化。
5. 将 Prometheus 查询结果沉淀为资产、目标和告警业务对象。
6. 增加工单状态流转接口，例如接单、转派、提交处置、关闭、备注。
7. 接入 FastAPI AI 服务、大模型 API、向量数据库和知识库问答。
8. 增加 ECharts 趋势图、详情抽屉、筛选条件和告警规则管理。
